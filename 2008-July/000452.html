On Wed, Jul 02, 2008 at 10:40:03PM +0200, Oliver Hartkopp wrote:
> Robert Schwebel wrote:
> > On Wed, Jul 02, 2008 at 09:52:01PM +0200, Wolfgang Grandegger wrote:
> >   
> >> You can either set the bit-rate *or* the bit-timing. If you set the 
> >> bit-rate, the bit-timing will be determined by an algorithm or a 
> >> built-in table. Therefore the correct wording should have been: "The 
> >> indirect setting via updatable table seems not that attractive".
> >>     
> >
> > In our approach, you *specify* the bit timing and *call*it 125000. Then
> > you *select* the 125000 "bitrate".
> >
> > You can also specify a bit timing and call it Fridolin_Tux. It could
> > then be selected by
> >
> > 	ifdown can0
> > 	canconfig can0 Fridolin_Tux
> > 	ifup can0
> >
> > This is completely independend of any heuristics (which could also be
> > triggered to specify such a line). You could add an API for "please
> > invent a bit timing for 125000" to trigger an algorithm, or a "please
> > generate a CiA-well-known-bit-timing for 250000" call. The end result is
> > always a bit timing set, which is indexed by the "speaking name" which
> > reflects the bitrate.
> >
> >   
> 
> ACK :-)
> 
> > This is convenient for users, because they can set the bitrate in an
> > intuitive way (canconfig can0 125000). The system administrator / BSP
> > integrator has to know what 125000 means on this particular hardware
> > (chip + board + corresponding clocks) and has to configure it somewhere.
> > And the end user can define his very own stuff if he knows what he does.
> >
> >   
> 
> ACK.
