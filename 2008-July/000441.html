On Wed, Jul 02, 2008 at 09:36:28AM +0200, Wolfgang Grandegger wrote:
>>> The second method would be nice if the setting would be hardware
>>> independent. Unfortunately, that's not the case and therefore I see no
>>> real benefit using a hardware independent format.
>>
>> Hmm? I don't understand what you mean.
>
> A bit-time stetting for one CAN controller may not work for another one,
> hence the setting is hardware dependent. You agree?

For one controller you may need to provide a certain set of the CiA
Model Values (which are hardware dependend, yes). But the data format is
standard and *not* hardware dependend, that makes a big difference.

> > Note, however, that Linux is no democracy. We want the *best*
> > solution, not the one most people vote for.
>
> And who decides what the best solution is?. Fact is that most people
> would implement it differently :-(.

Nonono, if we still do not agree on the *right* way, we may need to
either find more concrete use cases or somehow haven't seen all the
corner cases.

With Juergen's original patch, we have shown that the generic, hardware
independend timing model solves the problem that different controllers
have to be parametrized differently, in order to get what the system
designer supposes to be the right timing. Until now, I still havn't seen
an argument for a case where it does *not* work.

So either I havn't understood the problem or there is no problem or we
need to collect more data :-)
