Robert Schwebel wrote:
> On Wed, Jul 02, 2008 at 09:32:20PM +0200, Wolfgang Grandegger wrote:
>>> For one controller you may need to provide a certain set of the CiA
>>> Model Values (which are hardware dependend, yes). But the data format
>>> is standard and *not* hardware dependend, that makes a big
>>> difference.
>> To be more precise, the settings depend on the CAN clock used, apart
>> from the some very strange CAN controllers using non-standard ranges
>> for the time phases.
> 
> Ok, so for one hardware platform.
> 
>>> So either I havn't understood the problem or there is no problem or
>>> we need to collect more data :-)
>> There is no problem. Most people voted for the hardware independent
>> format. Let's go for it. But the indirect setting via table seems not
>> that attractive.
> 
> How do you handle the case then that users want to switch bitrates? It
> doesn't work without tables.
> 
> 	canconfig can0 bitrate 125000
> 
> Where does the timing information for 125000 come from, if not from a
> table?

You can either set the bit-rate *or* the bit-timing. If you set the 
bit-rate, the bit-timing will be determined by an algorithm or a 
built-in table. Therefore the correct wording should have been: "The 
indirect setting via updatable table seems not that attractive".
