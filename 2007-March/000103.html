Oliver Hartkopp <socketcan at hartkopp.net> writes:

> So for me there could be two ways to solve this problem:
> 
> 1) Create different mailing lists and indicate clearly which problem
> class is handled inside which ML, so that every user can post and
> discuss according his special field of interest.
> 
> 2) Use exact ONE mailing list to inform/discuss on the socketcan
> project and it's technical approaches in all regarded
> ISO-layers. This ML would also catch the SVN commit mailings (on
> which you may also start a technical discussion).

The commit mailing list should stay separate and it should only
receive the automatically generated commit messages from SVN, nothing
else.  Also, the SVN log messages should not be misused as
announcements for new functionality/changes checked into the repo.
The log message should be very short and concise, describing only the
change made, and maybe give a short (one line or so) reason for the
change.  It should also not discuss advantages/disadvantages or future
possible developments and the like.  This belongs to one of the
discussion mailing lists.

I would further vote to have two ML, socketcan-users and socketcan-devel,
which IMO more clearly shows the purpose of the two lists.  Most of
our previous discussions would take place on the devel ML,
announcements of changes and new releases would go to both.  I would
expect the users ML to be very low-volume traffic, but this could
change when the code moves into the mainline kernel and hopefully gets
more users.

Nevertheless, merging all discussion lists into one and having a
separate SVN commit list would be also OK, IMO.
