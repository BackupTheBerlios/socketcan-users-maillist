Hi Robert,

i appreciate your comment as i am quite unlucky about the current 
mailing list situation.

Since the socketcan-core mailing list had been created, the 
socketcan-users ML is 'very calm'. The highest activity has been on the 
socketcan-commit ML. User relevant topics e.g. "Magics of message 
filtering" have been posted on the socketcan-core list [i would assume 
this topic in socketcan-users]. And user relevant fixes and new 
functionalities have only been posted on the socketcan-commit list in 
svn logs [a relevant information for users, if they should update their 
local svn working copy]. Providing a cumulative project status on 
socketcan-users like i did in 
https://lists.berlios.de/pipermail/socketcan-users/2007-March/000095.html 
cannot be the right way to inform the users that do not listen to 'every 
interesting' mailing list ...

So for me there could be two ways to solve this problem:

1) Create different mailing lists and indicate clearly which problem 
class is handled inside which ML, so that every user can post and 
discuss according his special field of interest.

2) Use exact ONE mailing list to inform/discuss on the socketcan project 
and it's technical approaches in all regarded ISO-layers. This ML would 
also catch the SVN commit mailings (on which you may also start a 
technical discussion).

Indeed regarding the traffic & requests on the socketcan project i would 
tend to (2) - using exact one mailing list (e.g. the mostly populated 
socketcan-users ML) to get in touch with *all* relevant project information.

So please (users & mailing lists owners) vote for your favourite layout 
by replying "1" or "2" or contribute new ideas ...

Best regards,
Oliver
