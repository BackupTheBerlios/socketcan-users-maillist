Which is quite essential if you *really* want to use it - and we use it
in *production* systems. Until now, it always was a huge mess to bring
SocketCAN with all necessary features to a random modern and non-x86
kernel.

> IMO it's quite useless to maintain a second 'cansend' or 'candump' or

It's like with any other OSS projects out there. Things get done if
somebody needs them. If you need something integrated in the canutils,
please send patches. Otherwhise we'll do it if we need it of if one of
our customers needs it.

> It would be nice, if you could answer this thread:
> https://lists.berlios.de/pipermail/socketcan-users/2008-October/000523.html

The network layer alone is absolutely irrelevant. For production
projects, you need the *whole* stack, including bittime setting, being
able to work on random non-release kernels etc and all this is
extraordinarily complicated until now and was the reason for Marc to
move to git.

Yes, there is space for improvement for the canutils and I second your
oppinion that things should be unified. So please feel free to send
patches, or wait unitl it went high enough on our pet project todo list
that somebody actually starts working on it :)
