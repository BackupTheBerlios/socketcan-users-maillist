Daniel Schnell wrote:
> On Mon, 2008-10-27 at 15:21 +0100, Wolfgang Grandegger wrote:
> ...
>>> We really don't like to use a newer version of the linux kernel at the
>>> moment, so backporting socketcan is the prefered method. Unfortunately
>> What a pity.
> 
> Well, yes, but in my experience new kernels on powerpc tend to break the
> oftree and driver bindings for the oftree quite happily. Until this API
> is not settled yet, it's a tedious work chasing up.

Well, upgrading might indeed be a pain as various interfaces changed
substantially like I2C, etc. But The latest and greatest Socket-CAN is
also available for 2.6.24 and the API definition has settled in the
meantime.

> ... 
>> Setting CAN device parameters like the bit-rate is now supported via
>> SYSFS files:
>>
>>   $ echo "125000" > /sys/class/net/can0/can_bittiming/bitrate"
> 
> Okay, worked.
> 
>>   $ ifconfig cano up
> 
> Worked as well !
> 
> But without
> echo "125000" >/sys/class/net/can0/can_bittiming/bitrate
> 
> I got
> ifconfig: SIOCSIFFLAGS: Invalid argument

That's by intention. A default bitrate is error-prone.

> So the former is essential and I have to change some startup scripts.
> Would have maybe been good to provide a compatible canconfig program for
> backwards compatibility ...

canconfig was never part of the Socket-CAN distribution (SVN repository).

> Is setting of the baudrate via user space API still supported ? 
> When upgrading socket-can, do I have to recompile all user space
> programs as well ?

The IOCTLs have been removed as there is little chance to get them
included in the mainline kernel. Please use the SYSFS files instead.

Also be aware that the bit-timing algorithm changed calculating somehow
different values for your MPC5200 board.

Wolfgang.
