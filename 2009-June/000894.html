>> I have found that my socketcan system is much more stable if I do not define SLC_CHECK_TRANSMIT 
> 
> Only the slcan driver uses such an option.
> 
>> I prefer to have the system generate a basic error, rather than get a kernel panic, when I try to send a message to a bus that is not properly terminated.  I suspect you were trying to boot up your environment, and your CAN bus was not fully connected with proper termination.  This was biting me all the time.  Once I undefined SLC_CHECK_TRANSMIT, all my kernel panics went away.  With SLC_CHECK_TRANSMIT defined, there is a path thru the socketcan driver which does result in the panic you are reporting. I don't claim to know what that path is, just that with SLC_CHECK_TRANSMIT set, you do get the panic you report, and with it undefined, the problem goes away.
> 
> A TX timeout will report an oops once but it will not panic or hang the
> system. For me this sounds like a problem/bug in the slcan driver.

But as TX timeouts definitely create several problems, i don't have objections
to #undef SLC_CHECK_TRAMSMIT by default.
