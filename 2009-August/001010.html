>>> By the way, its not a good idea
>>> to let others than the application
>>> decide how to handle a bus-off condition.
>> What do you mean with others? The bus-off can be handled by "another"
>> application or server. Or what exactly do you mean.
> 
> Sorry, I meant the auto-bus-on, automatic recovery of the bus-off state.
> At least the application should get an error indication and should be able to 
> stop the automatic recovery from bus-off after a certain number of retries.
> Bus-off can be caused by wrong bit time settings or  defective CAN 
> controller/transceiver. In this case auto recovery always will disturb the 
> running network if it tries to go bus-on again and again. 

The application can receive error messages and state changes via error
frames as described here:

http://lxr.linux.no/linux+v2.6.30/Documentation/networking/can.txt#L426

The automatic bus-off recovery can be disabled by setting restart-ms to
0 via netlink interface. Unfortunately, it currently requires stopping
the device before restart-ms can be reset but that could be changed if
there are strong reasons. An alternative solution is to handle the
recovery solely by the application (with restart-ms0 always disabled).
