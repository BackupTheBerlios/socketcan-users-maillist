simon.kallweit at intefo.ch wrote:
> I have now ported the c_can driver to the socketcan trunk. So far it 
> compiles and sort of works.

Good news :-)

>  I got a few questions though:
>
> (..)
> * There is a candev module in the socketcan trunk. I adapted the driver 
> to use the candev module. Is this the right thing to do? Also I've seen 
> that the candev driver is not tagged with a mod license, so it 'taints' 
> the kernel when loading.
>
>   

Indeed the candev library module's intention is to provide a general
functionality that's needed by CAN network devices. But it's worth
discussing if we would need an extra abstracted 'struct can_device' like
ethernet_device with a bunch of ND2CAN and CAN2ND macros and typedefs
like can_mode_t, etc.

Additionally the current ioctl() interface needs to be checked. I'm not
that sure, but i heard something about the removal of ioctls for
netdevices ...

These details would lead to unwanted feedback when mainlining the driver
code. So the candev.ko is a very good idea but under construction and
discussion in my opinion.

Maybe your request restarts the discussion about this open topic.

Btw. i'll add the MODULE_LICENSE() macro to remove the kernel tainting
messages as the code is already under GPL.

Oliver
