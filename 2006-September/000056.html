Yep!

The socket-can idea brings to you:

- Multiuser capabilities (try this with a character device)
- A socket-api that smells similar to PF_INET sockets (known usage for 
standard-IT programmers)
- A standardized network-device interface for CAN-Devices (not the 
ioctl(), ASCII, binary, whatever mix you mostly get when you buy a new 
CAN-Hardware and get a 'special driver' for it)
- It makes the CAN-Bus a real 'network' under Linux
- Transparent CAN transport protocol support (e.g. in vehicle diagnosis 
you have a point-to-point communication via CAN. The socket-can makes 
connections to an engine-ECU as easy as a connection via TCP/IP-sockets)
- you can have multiple applications (e.g. multiple candump's with 
different filter settings) on the same CAN-Bus
- you can have applications on real or virtual CAN-busses talking to 
each other on the same machine

Is this enough so far?
